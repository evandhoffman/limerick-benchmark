## What The Artefacts Show

### 1. `qwen3.5:35b-a3b-coding-mxfp8` is winning by doing less

Median trace and timing signals are materially smaller than the other two
models:

- median `wall_seconds`: 15.45
- median `first_edit_seconds`: 14.05
- median `trace_line_count`: 65
- median `duplicate_output_attempt_count`: 3
- median verification/self-correction markers: 0 / 0

This model typically writes one file once, near the end of the run, and stops.
That is exactly what you want from Aider whole-file mode: minimal chattiness,
minimal output repetition, and a fast final write.

Its code shape is mostly stable but not frozen. Eight runs used
`render_template_string`, while two returned raw HTML directly. The report’s
`Distinct solution shapes = 2` is real, but the variation is shallow; all runs
still converge on the same successful “read `limericks.txt`, split on `\n\n`,
render one random limerick with a refresh tag” pattern.

### 2. `gemma4:e4b` is stable, but slower for no visible payoff

`gemma4:e4b` lands in a narrow band:

- median `wall_seconds`: 21.45
- median `first_edit_seconds`: 20.0
- median `trace_line_count`: 130
- median `duplicate_output_attempt_count`: 36

The generated `app.py` is straightforward Flask with one route and inline HTML.
It does not use `render_template_string` in this batch, and it shows only two
distinct hashes across ten runs with one stable solution shape. That is good
repeatability, but it is slower than Qwen 3.5 without compensating quality in
the benchmark artefacts.

### 3. `qwen3.6:35b-a3b-coding-mxfp8` is reliable but expensive

This model passes every run, but its behaviour is visibly heavier:

- median `wall_seconds`: 33.95
- median `first_edit_seconds`: 32.4

- median `duplicate_output_attempt_count`: 69.5
- median verification/self-correction markers: 4 / 3

That is the same one-route successful app shape as Gemma, not a more ambitious
solution. The extra time is mostly thought/output churn before the final file
lands. Four distinct `app.py` hashes across ten runs reinforces the same point:
the model is exploring more surface area than the task needs.

## Aider-Specific Behaviour

All three models show the same high-level pattern:

- `files_created_count = 1` on every run
- `files_modified_count = 0` on every run
- `first_edit_seconds` is usually within about 1–1.5 seconds of
  `agent_seconds`

That means the models are not iterating through multiple code revisions inside
the workspace. They are effectively “thinking in the chat transcript” and then
dropping a single whole-file `app.py` near the end. For Aider benchmarking,
that is useful context: most of the wall time here is pre-write deliberation,
not file-edit iteration.

not treat that as a genuine order effect yet.

The per-model medians also do not point to a single universal “bad” position:

- `qwen3.5:35b-a3b-coding-mxfp8` is fastest at position 3
- `qwen3.6:35b-a3b-coding-mxfp8` is fastest at position 3

So the current position summary is useful as a descriptive view, but it is not
strong evidence of thermal or scheduling drift by itself.

## Bottom Line

If the goal is “best local coding model for this harness on this task,” the
latest run strongly favors `qwen3.5:35b-a3b-coding-mxfp8`.

It is:

- the fastest model
